DNS DoH Security Research RFC 8484 SSRF

What I Found Scanning the 4 Biggest Public DoH Resolvers

April 13, 2026 · 6 min read · Shannon Agarwal
DoH
4 Resolvers Scanned
5 Total Findings
2 High Severity
5 Security Plugins

DNS-over-HTTPS (DoH) was designed to fix DNS privacy:encrypting queries that were historically sent in plaintext. But encryption alone doesn't mean the resolvers handling those queries are implemented securely. I built ZeroDoh-Scanner, a plugin-based CLI tool that probes DoH endpoints for vulnerability classes defined in RFC 8484 and beyond, and ran it against the four most widely used public resolvers: Cloudflare, Google, Quad9, and NextDNS.

Here's what I found.

The Setup

ZeroDoh-Scanner runs five plugins against each resolver:

  • spec_violations — RFC 8484 compliance: malformed wire format, wrong Content-Types, oversized payloads
  • info_leakage — server header fingerprinting, internal IP disclosure in DNS answers
  • cache_poison — TTL consistency across repeated queries, DNSSEC validation gaps
  • rebinding — zero-TTL record handling, PTR responses for loopback addresses
  • ssrf — whether the resolver will resolve loopback, link-local, and cloud metadata hostnames

All checks are passive or semi-passive — no exploitation, just probing behavior directly observable from the DNS response layer.

Results

Cloudflare (1.1.1.1)

No findings. Cloudflare correctly rejects malformed payloads, strips internal IPs from responses, and refuses to resolve SSRF-relevant hostnames. This is what a hardened DoH implementation looks like.

NextDNS

Also no findings. NextDNS passed all checks cleanly, notable given it's a smaller operator compared to the hyperscalers.

Google Public DNS

[Medium] Accepts malformed DNS wire format Google's resolver accepted a request containing a structurally invalid DNS wire format payload without returning an error. RFC 8484 §4.1 specifies that a server receiving a malformed message MUST return a FORMERR (Format Error) RCODE. Instead, Google processed the request.

This matters because lenient parsing can be a signal of deeper implementation gaps. A parser that overlooks malformed inputs in testing may fail unpredictably on crafted inputs in adversarial conditions. It's a small surface, but it's spec non-compliance on a resolver handling hundreds of billions of queries per day.

Quad9

Quad9 was the most interesting resolver to scan, with findings across three vulnerability classes.

[High] Returns private/internal IP address When querying certain hostnames, Quad9 returned responses containing private RFC 1918 address space in DNS answers. Internal network topology can be inferred from DNS responses that should never include private IPs from a public resolver.
[High] Resolves SSRF-relevant hostname: localhost Quad9 returned a valid DNS response when queried for localhost. A hardened public resolver should refuse to resolve hostnames that map to loopback, link-local, or cloud metadata addresses. If an application uses a DoH resolver in its DNS resolution pipeline and that resolver answers for localhost, an attacker who can influence the hostname being resolved can potentially reach internal services:a precondition for SSRF via the DNS layer.
[Medium] TTL increased between repeated queries Repeated queries for the same record returned TTL values that increased between queries rather than counting down. Standard caching decrements TTL toward expiry. A TTL that increases suggests inconsistent caching across resolver nodes:relevant to cache poisoning research, where inconsistent cache boundaries give unpredictable feedback to birthday-attack-style probing at the DoH layer.
[Medium] Returns PTR records for loopback addresses Quad9 returned PTR records for reverse lookups on loopback address space (e.g., 1.0.0.127.in-addr.arpa). A public resolver shouldn't be authoritative for or willing to answer PTR queries in loopback space. Together with the localhost resolution finding, this suggests Quad9's filtering of reserved address space is incomplete.

What This Means

DoH adoption has grown significantly since RFC 8484 was published in 2018. Browsers, operating systems, and applications are increasingly routing DNS queries through these resolvers:often without the user's explicit awareness. That makes the security posture of the resolver itself critical infrastructure.

Cloudflare and NextDNS demonstrate that clean implementations are achievable. The Quad9 findings:particularly localhost resolution and private IP disclosure:are worth investigating further. Neither is a critical vulnerability in isolation, but together they represent incomplete filtering of reserved and sensitive address space on a resolver handling billions of queries per day.

Run It Yourself

ZeroDoh-Scanner is open source. Scan any DoH endpoint:

pip install git+https://github.com/shannon-encrypt/zerodoh-scanner.git
zerodoh scan --all --output html --out report.html
