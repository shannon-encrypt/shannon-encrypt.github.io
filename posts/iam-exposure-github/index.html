IAM Exposure on GitHub: What AI-Assisted Dorking Found in the Wild

March 22, 2026 · 8 min read · Shannon Agarwal
0xSEC // research edition

I built a tool that searches GitHub for exposed credentials. Here's what it found in 30 minutes, and what happened when one of them turned out to be real.

This research uses an automated GitHub dorking pipeline to surface IAM credential exposure across public repositories. Each finding gets enriched by the Claude API: severity score, identity provider fingerprint, MITRE ATT&CK mapping, and remediation guidance. The goal wasn't just to find exposed secrets. It was to understand how well AI-assisted triage performs at scale, where it breaks down, and what the full responsible disclosure loop looks like in practice.

Responsible disclosure note Repository names and organization identifiers have been omitted from this post except where a finding was confirmed real and reported. No credentials were tested, validated, or used in any way. One confirmed live credential set was reported to GitHub before this post was published.

01. Why GitHub Is Worth Searching

GitHub is the world's largest public code repository. Developers routinely commit .env files, service account keys, SAML metadata, and SSO configuration. Sometimes accidentally, sometimes because they're following a tutorial that modeled bad habits.

Attackers already run these searches manually and at scale. The question this research asks is: can a small AI-augmented pipeline surface the same findings faster, classify them more reliably, and do it responsibly?

The answer is mostly yes, with one critical caveat that almost buried the most important finding in the dataset.

02. How the Pipeline Works

Dork queries are organized into four categories targeting known patterns of IAM exposure:

  • Cloud credentials: AWS access keys, GCP service account JSON, Azure client secrets
  • Okta / SAML / SSO: Okta API tokens, SAML metadata files, SSWS tokens
  • Microsoft Entra / Azure AD: service principal credentials, tenant/client ID pairs
  • Generic secrets: .env files, secrets.yml, .dockercfg, .npmrc auth tokens

25 queries run via the GitHub Search API, capped at 10 results per query to stay within rate limits. Results are deduplicated by repo and file path. Raw file content is processed in memory only. A redaction function replaces credential values with [REDACTED] before anything is stored or sent to the AI layer. Each finding then goes to Claude with a structured classification prompt.

The model (claude-sonnet-4-6) returns a severity score, a likelihood judgment on whether the credential is active vs. test data, the identity provider, applicable MITRE ATT&CK technique IDs, and remediation guidance. Every high and medium severity finding gets manually reviewed by opening the source file directly.

AI enrichment requires an Anthropic API key and is optional — the tool runs without one using --skip-enrichment, which produces a raw report of unclassified findings instead.

That last step turned out to be the most important part of the whole pipeline.

04. The AI Got "Likely Real" Almost Completely Wrong

This is the most important finding in the research, and it almost didn't make it into the write-up.

The Claude API flagged all 30 findings as is_likely_real: true. At first glance that sounds like the tool is working. Manual review of the high-severity findings revealed a consistent failure mode: the model was classifying based on structural pattern matching, not actual credential value presence.

Two clear examples from the high-severity queue:

Example 1: Shell script loader flagged as HIGH A file containing export OPENAI_API_KEY="${key_base_dir}/openai.key" was classified as a live credential exposure. In reality it's a shell script that loads keys from local files on the developer's machine at runtime. No credential values are present in the file at all, only variable references to local paths.
Example 2: Vercel secret references flagged as MEDIUM A now.json deployment config containing "MELI_CLIENT_SECRET": "@meli-manager-meli_client_secret" was flagged as a live credential. The @ prefix is Vercel's syntax for a secret reference. The actual value lives in Vercel's secrets vault and is never written to the file.

In both cases the AI correctly identified the types of credentials, the identity provider, and generated accurate remediation guidance, but couldn't distinguish between a file that contains a secret and one that references a secret stored elsewhere.

This is a meaningful limitation for any team building AI-assisted security tooling. The model sees the structure and the key names. It can't see what isn't there. Human review of every high/critical finding is not optional. It's the whole game.

05. The One Real Finding

After manually reviewing the high-severity queue, one finding was unambiguously real.

A public repository's .env file contained a live Okta SSWS API token for a named Okta organization, a MongoDB Atlas connection string with username and password in the URL, and an Okta Workflows API endpoint with a bearer token. The SSWS token format was valid, the Okta org domain was a real organization and not a dev- sandbox, and the MongoDB connection string pointed to a live Atlas cluster.

It was reported to GitHub before this post was published. The file URL was provided so the owner could take action. No credentials were tested or used.

The complete research loop Build a tool. Find a real exposure. Report it responsibly. Document the methodology. The point is not to name the organization or demonstrate impact. It's to show that this class of exposure exists, is findable with modest tooling, and has a responsible path forward.

06. Key Observations

Educational repositories are a systemic risk vector. One of the HIGH findings came from a publisher's repository for a widely-read book on Python microservices, with 400 stars, actively used as a learning reference. The repo contained what appeared to be a real Okta dev instance URL and client credentials committed alongside tutorial code. If the example commits credentials, so will the readers.

SAML metadata exposure is underappreciated. 14 of 30 findings involved SAML or SSO configuration. SAML metadata files aren't credentials in the traditional sense, but they expose entity IDs, assertion consumer service URLs, X.509 signing certificates, and SSO endpoint structure. For a targeted attacker that's reconnaissance that would otherwise require active probing. Two findings involved government or large institutional infrastructure.

The .env file problem isn't getting better. Despite years of guidance, .gitignore templates, GitHub secret scanning, and widespread documentation, developers continue to commit .env files containing production credentials. Several findings were from repositories pushed within the last 12 months. This is not a legacy problem.

Credential age doesn't equal credential safety. Several findings came from repositories last pushed in 2017-2019. Old credentials are not necessarily rotated credentials. Without an active rotation policy or an alert that forces remediation, exposed secrets can sit in public repositories for years unnoticed.

07. Detection Engineering Takeaways

If you're running Splunk or building detections, these findings map directly to actionable rules:

  • Alert on code pushes containing AKIA[A-Z0-9]{16}: the AWS access key format is fixed and highly detectable
  • Enable GitHub Advanced Security secret scanning, free for public repos, covers 200+ credential patterns natively
  • For Okta environments: alert on API token creation events (system.api_token.create) and audit token scopes quarterly
  • Monitor for .env, secrets.yml, .dockercfg, and serviceAccountKey.json in commit diffs via GitHub webhooks or your SIEM's code repository integration
  • For Azure/Entra: alert on service principal credential additions outside your approved provisioning workflow
Highest-ROI action Enforce .gitignore templates at repo creation and run GitHub's secret scanning on all public repositories. It's free and it works.

08. Limitations

GitHub's search API returns a ranked sample, not exhaustive results. 30 findings from 25 queries represents a floor, not a ceiling. Credential validity was not verified. Severity scores are based on structural analysis and AI judgment, not live testing. The AI classification prompt wasn't fine-tuned for this use case; a purpose-built classifier would likely perform better on the value-vs-reference distinction that caused most false positives. Rate limiting also constrained query volume. A longer research window would surface more findings.

Final thoughts

The most valuable thing this pipeline did wasn't find the real credential. It was surface it fast enough that I could report it before anyone else noticed. The AI handled the volume; the human judgment handled the call on what was actually real.

That division of labor is the right model for this kind of work. AI triage at scale, human review at the edges, and a clear responsible disclosure path before anything gets published.

The source code, dork query library, and full methodology are available in the companion GitHub repository. If this research is useful, the most important thing you can do is enable GitHub secret scanning on your organization's repositories today.
